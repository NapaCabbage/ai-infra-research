# GPU 硬件架构：从芯片到核心

## 1. CUDA 统一计算架构

GPU 采用 **CUDA（Compute Unified Device Architecture）** 架构，这是一种简化的统一设计理念。与早期 GPU 的固定管线架构不同，现代 GPU 配备了高度同质的硬件单元——**Streaming Multiprocessors（SMs）**——能够执行广泛的计算任务。

这一统一设计带来的核心优势：
- 软硬件更容易匹配
- 硬件工程师不再需要猜测各管线阶段的负载比
- 更灵活的计算能力

## 2. 芯片层级结构（自上而下的视角）

### 2.1 完整的 GPU 芯片

```
┌─────────────────────────────────────────┐
│         H100/H800 GPU 芯片              │
│     (132 个 Streaming Multiprocessors)  │
│                                         │
│  ┌──────────┐  ┌──────────┐ ... ┌──────────┐
│  │   GPC 0  │  │   GPC 1  │     │  GPC 11  │
│  │ (多个TPC)│  │ (多个TPC)│     │ (多个TPC)│
│  │ (多个SM) │  │ (多个SM) │     │ (多个SM) │
│  └──────────┘  └──────────┘ ... └──────────┘
│                                         │
│  ┌─────────────────────────────────┐   │
│  │      高带宽内存（HBM）          │   │
│  │       80 GiB RAM / H800          │   │
│  └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

### 2.2 GPC（Graphics/GPU Processing Cluster）

**GPC** 是 GPU 芯片的二级划分单元，包含：
- 多个 **TPC（Texture Processing Clusters）**
- 光栅化引擎（用于图形，AI 推理通常不需要）

在 Compute Capability 9.0 及以后（H100/H800），GPC 引入了一个新的线程层级概念：**thread cluster**（线程簇），使多个线程块能调度到同一 GPC 上，实现细粒度的分布式共享内存（distributed shared memory）。

**为什么 GPC 很重要？** 在 DeepSeek-V3 的例子中，模型利用 SMs 之间的通信来实现专家并行。GPC 是硬件上的逻辑分组，为这种通信提供了优化基础。

### 2.3 TPC（Texture Processing Cluster）

TPC 是 GPC 内的子单元，其本身是 SM 的集合。这种层级化设计使 GPU 能够在多个粒度上组织并发执行。

### 2.4 SM（Streaming Multiprocessor）

SM 是 GPU 的核心执行引擎，类似于 CPU 核心但有重要区别。

**H100/H800 的 SM 配置：**
- 总数：132 个 SM
- 每个 SM 的能力：
  - 4 个 Warp Scheduler（每个可调度一个 warp）
  - 每个 warp 包含 32 个线程
  - 每个 SM 可并行执行 128 个线程（4 × 32）
  - 总并发线程数（H100）：> 250,000 个（132 SM × 2,048 并发线程/SM）

**SM vs. CPU 核心的对比：**

| 特性 | GPU SM | CPU 核心 |
|------|---------|---------|
| 线程并行执行 | ~16K+ 线程 | ~384 线程（AMD EPYC） |
| 功耗效率 | ~5 cW/线程 | ~1.25 W/线程 |
| 上下文切换 | ~1 纳秒（时钟周期） | ~微秒级（数百~数千周期） |
| 缓存策略 | 程序员管理为主 | 硬件自动管理 |
| 算力分配 | 芯片面积主要用于计算 | 面积分配给缓存、控制、预测 |

## 3. SM 内部架构

### 3.1 核心层级结构

```
┌────────────────────────────────────────┐
│   Streaming Multiprocessor (SM)        │
│                                        │
│  ┌──────────────────────────────────┐  │
│  │   128 CUDA Core                  │  │
│  │   (32 FP32 + 32 INT32 per lane)  │  │
│  │                                  │  │
│  │   4 × Tensor Core                │  │
│  │   (1 per Warp Scheduler)         │  │
│  └──────────────────────────────────┘  │
│                                        │
│  ┌──────────────────────────────────┐  │
│  │  4 × Warp Scheduler              │  │
│  │  (每个调度 32 个线程/周期)       │  │
│  └──────────────────────────────────┘  │
│                                        │
│  ┌────────────────────────────────┐   │
│  │  Register File（寄存器堆）     │   │
│  │  (快速，~1ns 访问延迟)         │   │
│  └────────────────────────────────┘   │
│                                        │
│  ┌────────────────────────────────┐   │
│  │  L1 数据缓存（256 KiB）        │   │
│  │  (程序员管理)                  │   │
│  └────────────────────────────────┘   │
│                                        │
│  ┌────────────────────────────────┐   │
│  │  Load/Store Units (LSU)        │   │
│  │  Special Function Units (SFU)  │   │
│  │  Tensor Memory Accelerator     │   │
│  └────────────────────────────────┘   │
└────────────────────────────────────────┘
```

### 3.2 CUDA Core（CUDA 核心）

**定义：** CUDA Core 是执行标量算术指令的 GPU 核心。

**H100 配置：**
- 每个 SM：128 个 FP32 CUDA Core
- 全芯片：132 SM × 128 = 16,896 CUDA Core

**特点：**
- 32 个 CUDA Core 分组为一个 warp，由同一 Warp Scheduler 控制
- 组内线程执行相同指令，但操作不同寄存器（SIMD）
- 不同 SM 架构中，CUDA Core 的组成可能不同（FP32、INT32、FP64 混合）

**与 AI 训练/推理的关系：** CUDA Core 处理激活函数、归一化等标量操作；Tensor Core 处理矩阵乘法主体运算。

### 3.3 Tensor Core（张量核心）

**定义：** Tensor Core 以单条指令执行整个矩阵操作，是现代 AI GPU 的性能引擎。

**H100 配置：**
- 每个 SM：4 个 Tensor Core（每个 Warp Scheduler 1 个）
- 全芯片：132 SM × 4 = 528 个 Tensor Core
- 性能：**提供比 CUDA Core 高 100 倍的浮点运算吞吐量**

**工作原理（以 HMMA16.16816.F32 为例）：**

```
单个 Tensor Core 指令：D = AB + C
  - 输入：两个 16 位半精度矩阵（A: 16×8, B: 8×16）
  - 输出：FP32 单精度累积结果
  - 单指令内的运算：16 × 8 × 16 = 2,048 MAC（乘加）操作
  - 摊算到 32 线程 warp：64 MAC/线程/指令
```

**FP8 精度与累积精度的关键点：**

根据 DeepSeek-V3 的观察，Tensor Core 的 FP8 累积精度约为 14 bit。这是因为：
- FP8 输入在计算过程中逐步累积到更高精度（通常 FP32 或中间格式）
- 累积误差导致有效精度约为 14 bit
- 这对大规模矩阵乘法仍可接受，但需要谨慎处理

**内部架构推测：** Tensor Core 可能采用收缩阵列（systolic array）设计，类似于 TPU，但具体内部设计保密。

### 3.4 Warp Scheduler（Warp 调度器）

**定义：** Warp Scheduler 决定每个时钟周期执行哪个 warp（32 个线程的组）。

**关键机制：隐藏延迟（Latency Hiding）**

```
SM 的超快速上下文切换：

CPU 线程上下文切换：
  时间成本：微秒级（数百~数千个时钟周期）
  原因：需要保存/恢复所有上下文
  后果：缓存行为变差，性能下降

GPU warp 切换：
  时间成本：1 个时钟周期（~纳秒）
  原因：每个线程有独立寄存器（无需转移），缓存可共享
  优势：当某 warp 等待内存时，立即切到另一 warp
```

**SM 的 4 个 Warp Scheduler 可以：**
- 独立调度 4 个 warp（共 128 个线程）
- 轮转调度多达 64 个 warp（2,048 并发线程）
- 在内存延迟（100+ 周期）时切换到其他就绪的 warp
- 使计算资源始终有任务可做

**与 DeepSeek-V3 的关联：** 20 个 SMs 用于通信的设计利用了 Warp Scheduler 的快速切换能力，使通信和计算的隐藏与重叠成为可能。

### 3.5 寄存器堆（Register File）

**定义：** 寄存器是 SM 内最快的存储，用于线程间的中间计算结果。

**配置：**
- 高速内存技术：比 L1 缓存快 ~10 倍
- 访问延迟：~1 纳秒
- 动态分配：32 bit 寄存器单元，可灵活分配为 FP32、INT32、FP16、FP8
- 上限：每个线程可分配多个虚拟寄存器，编译器优化分配

**寄存器压力（Register Pressure）的影响：**
- 高寄存器使用 → 并发线程数减少 → 占用率（occupancy）降低
- 低占用率 → 无法充分隐藏内存延迟 → 性能下降

### 3.6 L1 数据缓存

**定义：** SM 的片上 L1 缓存，实现为可编程的共享内存。

**H100 配置：**
- 每个 SM：256 KiB
- 全芯片：132 × 256 KiB = 33 MiB
- 延迟：~L1 缓存快，比 GPU RAM 快得多
- 技术：SRAM（与 CPU 缓存、寄存器相同）

**特点：**
- **程序员管理**：与 CPU 硬件自动管理不同，GPU 需要代码显式利用
- **片上 SRAM**：功耗高但速度极快
- 访问单元：Load/Store Units (LSU) 控制访问

### 3.7 Load/Store Unit (LSU) 和 Special Function Unit (SFU)

**LSU（负载/存储单元）：**
- 处理内存访问（从 L1 缓存和全局 RAM）
- 连接寄存器堆与内存系统

**SFU（特殊函数单元）：**
- 加速超越函数（exp, sin, cos, sqrt）
- 对神经网络中的激活函数关键
- 汇编指令：MUFU.SQRT, MUFU.EX2 等

### 3.8 Tensor Memory Accelerator (TMA)

**定义：** Hopper/Blackwell 专有硬件，加速多维数组的内存访问。

**功能：**
- 直接从 GPU RAM 加载数据到 L1 缓存，**绕过寄存器**
- 异步执行：单线程触发大量复制，然后继续其他工作
- 支持仿射寻址：`addr = width * base + offset`（矩阵访问的通用模式）

**优势：**
- 减少寄存器压力
- 减少算术带宽占用
- 异步模型支持生产者-消费者并发

## 4. 内存层级结构

```
┌────────────────────────────────────────┐
│      GPU 全局内存（GPU RAM）           │
│    80 GiB HBM（H100/H800）             │
│   带宽：~2 TB/s，延迟：数百周期        │
└────────────────────┬───────────────────┘
                     │
            各 SM 共享（全可访问）
                     │
┌────────────────────▼───────────────────┐
│        L1 数据缓存（每个 SM）           │
│    256 KiB，32 KiB × 8 分区             │
│   带宽：~TB/s 级，延迟：~5~20 周期     │
└────────────────────┬───────────────────┘
                     │
            线程块内共享（Shared Memory）
                     │
┌────────────────────▼───────────────────┐
│         寄存器堆（每个 SM）             │
│    每线程：多个 32 bit 寄存器           │
│   带宽：~极高，延迟：1 周期             │
└────────────────────────────────────────┘
```

**Tensor Memory（仅 Blackwell）：** 专用于 Tensor Core 输入/输出，独立于主寄存器堆。

## 6. AI 训练/推理中的关键作用

### 6.1 为什么 Tensor Core 主导性能？

```
矩阵乘法是神经网络的核心运算：
  - 前向传播：Y = W @ X
  - 反向传播：dW = dY @ X^T, dX = dY @ W^T

单个 Tensor Core 指令 = 2,048 MAC
单个 CUDA Core 指令 = 1 运算

吞吐量差异：100 倍

因此，高效 Tensor Core 使用决定了大模型训练速度。
```

### 6.2 DeepSeek-V3 利用 20 个 SMs 通信的含义

```
为什么需要 SMs 之间通信？
  - 专家并行（Expert Parallelism）
  - 集合通信操作（All-reduce, All-gather）
  - 动态负载均衡

为什么选择 20 个 SMs（从 132 个中）？
  - 通信开销 vs. 计算收益的权衡
  - 20 个 SMs = 20 × 128 = 2,560 并发线程
  - 足以隐藏全局同步延迟
  - 剩余 112 SMs 可用于计算
  - NVLink 带宽足以支持这个规模的通信

这体现了 GPU 架构的分层设计：
  - 芯片内部有多个并行通道（GPC/TPC/SM）
  - 不同 SMs 可独立工作，也可协调通信
```

### 6.3 延迟隐藏的重要性

```
GPU 计算延迟典型值：
  - L1 缓存访问：~5 周期
  - 全局内存访问：~300+ 周期
  - Tensor Core 依赖链：~数十周期

Warp Scheduler 的作用：
  每当一个 warp 等待数据时，立即调度另一个 warp 执行
  1 个 SM 可并发 64 个 warp
  关键指标：占用率（Occupancy）= 实际运行 warp 数 / 最大 warp 数
  高占用率 → 更好地隐藏延迟 → 更高吞吐量

与 CPU 对比：
  CPU 通过大缓存、指令预测隐藏延迟（开销大）
  GPU 通过大量并发线程隐藏延迟（更高能效）
```

## 7. 与已学内容的关联

### 关键观点整合

1. **DeepSeek-V3 的 20 SMs 通信**
   - 现在理解：SM 是独立的执行引擎，每个有自己的寄存器、L1 缓存
   - 132 个 SMs 可以通过 NVLink 互连
   - 20 个 SMs 的选择是工程权衡：足够的并发度 + 可管理的通信复杂度
   - 利用 Warp Scheduler 快速切换，使通信和计算重叠

2. **FP8 精度与 Tensor Core**
   - Tensor Core 内部以更高精度（可能是 FP32）累积
   - 累积误差使有效精度约为 14 bit（低于 FP8 的 8 bit）
   - 这是硬件设计的权衡：速度（超快的矩阵运算）vs. 精度
   - 对大规模模型的精度影响在可接受范围内

3. **Warp Scheduler 与延迟隐藏**
   - Warp Scheduler 每纳秒切换一个新的 warp
   - 4 个独立的 Scheduler 可管理 64 个 warp（2,048 线程）
   - 当前 warp 等待内存时，立即执行其他 warp
   - 这是 GPU 比 CPU 更能容忍高延迟内存的根本原因
   - 对神经网络训练至关重要（大量内存访问）

4. **SM 的统一设计**
   - 不同 GPU 型号的 SM 结构基本相同
   - 主要差异：Tensor Core 的规格、缓存大小、频率
   - 代码可跨多代 GPU 运行（兼容性强）
   - 但最优化需要考虑每一代的特性

5. **硬件支持的计算模式**
   - CUDA 编程模型直接映射到 SM 的物理结构
   - Thread Block → SM，Warp → Warp Scheduler，Thread → 寄存器
   - 充分理解硬件层级有助于写出高效内核
